## T10 — AuthProvider, garde de route, écran Connexion

### Objectif
Permettre à un membre de se connecter en email + mot de passe, rester connecté sur son poste, et
protéger toutes les routes de l'app.

### Décision clé
`AuthProvider` **au-dessus** de `DirectoryProvider` dans `main.tsx`. La session est déjà persistée par
le client Supabase (`persistSession`) : **ne pas réimplémenter** de stockage. La garde redirige vers
`/connexion` sans session, et de `/connexion` vers `/` si session. `DirectoryProvider` doit
**(re)charger quand la session change** — c'est la seule modification autorisée dans `src/data/`
(s'abonner à `onAuthStateChange` ou consommer l'état d'auth).

### Lire / Modifier
- Créer `src/features/auth/AuthProvider.tsx` : contexte `{ session, userId, member, loading, signIn, signOut, refreshMember }`.
  `member` = ligne `members` de l'utilisateur courant (pour le nom + le rôle).
- Créer `src/features/auth/RequireAuth.tsx` (ou équivalent) : garde de route.
- Remplacer le stub `src/features/auth/ConnexionPage.tsx`.
- Modifier `src/app/router.tsx` (garde), `src/app/Layout.tsx` (pastille profil : initiales du membre +
  menu « Mon profil » → `/membres`, « Se déconnecter »), `src/main.tsx` (`<AuthProvider>` au-dessus de
  `<DirectoryProvider>`), et **au besoin** `src/data/DirectoryProvider.tsx` (rechargement à la connexion).

### Étapes
1. `AuthProvider` : au montage, `getSession()` ; s'abonner à `onAuthStateChange` ; charger la ligne
   `members` correspondante (`select('*').eq('id', uid).maybeSingle()`).
2. `ConnexionPage` fidèle à la maquette : carte centrée (largeur 380, radius 20, fond `#fff`), pastille
   « M » en dégradé, « MSP Ménilmontant », « Réservé aux membres de la MSP », champ **email**, champ
   **mot de passe**, bouton « Se connecter » en dégradé pleine largeur. **Ajouts** (≠ maquette, cf.
   décision) : message d'erreur lisible en cas d'échec (« Email ou mot de passe incorrect »), état
   « connexion… » sur le bouton, et mention discrète « Mot de passe oublié ? Contactez le référent ».
3. Garde : pendant `loading`, ne rien rediriger (éviter un flash) ; puis rediriger.
4. Layout : initiales depuis `member.prenom/nom` (repli sur l'email si vides) ; menu déconnexion.

### Validation
- Auto (bloque le commit) : `npm run build` + `npm run typecheck` → 0 erreur.
- Humain (visuel/parcours) : connexion OK, session conservée après rechargement de page, redirection
  hors session, déconnexion. → consigner la checklist dans ce fichier (§Bilan).

## T11 — Écran Membres (liste, profil, mot de passe)

### Objectif
Afficher les membres de la MSP et permettre à chacun de compléter son profil et changer son mot de passe.

### Décision clé
**L'app ne peut pas créer de comptes** (cela exige la clé de service, jamais exposée au navigateur) :
le bouton « + Inviter un membre » de la maquette **n'appelle aucune API** — il n'est visible que pour un
`role === 'referent'` et ouvre un encart explicatif : « Créez le compte dans Supabase → Auth → Users →
Add user (email + mot de passe initial + Auto Confirm). La fiche membre est créée automatiquement. »
Le profil est important : sans `nom`/`prenom`, les commentaires s'afficheront avec un repli sur l'email.

### Lire / Modifier
- Remplacer le stub `src/features/membres/MembresPage.tsx`.
- Lecture seule : `src/data/DirectoryProvider.tsx` expose déjà `members`.

### Étapes
1. Liste fidèle à la maquette : carte par membre (`Avatar` initiales, nom, `profession · email`, badge
   « Référent » si `role==='referent'`).
2. Bloc « Mon profil » : champs `prenom`, `nom`, `profession` → `update` sur sa propre ligne `members`
   (la RLS n'autorise que la sienne) ; puis rafraîchir (`refreshMember` + `reload()` du directory).
3. Bloc « Mot de passe » : nouveau mot de passe + confirmation → `updatePassword` (aucun email envoyé).
4. Bouton « + Inviter un membre » : visible seulement si référent → encart explicatif (aucun appel API).

### Validation
- Auto : `npm run build` + `npm run typecheck` → 0 erreur.
- Humain : liste correcte, édition du profil persistée, changement de mot de passe effectif. → §Bilan.

### Garde de route et persistance de session

- `AuthProvider` lit `getSession()` au montage puis s'abonne à `onAuthStateChange` (les deux via
  `src/data/auth.ts`, non réimplémentés) ; il résout aussi la ligne `members` du membre courant. Pendant
  la résolution initiale, `loading=true` et personne ne redirige (ni `RequireAuth`, ni `ConnexionPage`)
  — évite le flash décrit dans T10 §Étapes 3.
- `RequireAuth` (autour de `<Layout>`, donc de toutes les routes protégées) redirige vers `/connexion`
  si `session` est `null` une fois `loading` retombé à `false`. `ConnexionPage` fait le trajet inverse
  (redirige vers `/` si une session existe déjà).
- La session elle-même reste gérée par le client Supabase (`persistSession` + `autoRefreshToken`, déjà
  configuré dans `src/lib/supabase.ts` à S2) : ni `AuthProvider` ni la garde ne touchent au stockage.
- `DirectoryProvider` (S2, non modifié) recharge le dataset annuaire quand l'uid change (connexion,
  déconnexion) mais ignore les évènements `TOKEN_REFRESHED` qui ne changent pas d'utilisateur.

### Écarts avec S7.md

- Aucun écart de périmètre. Seul ajustement mineur non prévu explicitement par le plan : le bloc « Mon
  profil » et « Mot de passe » de Membres écrit directement via `supabase.from('members').update(...)`
  et `updatePassword` depuis `MembresPage.tsx` (pas via `src/data/`, conformément à la contrainte « la
  seule modification autorisée dans `src/data/` est le rechargement de `DirectoryProvider` »).
