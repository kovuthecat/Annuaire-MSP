## T1 — L'extracteur (JSON-LD prioritaire, DOM en repli)

### Objectif
Depuis une page Doctolib ouverte, produire un objet conforme au **contrat `prefill`** de S1.

### Décision clé
- **Priorité JSON-LD** : lire les `<script type="application/ld+json">` de la page, parser le/les objets
  schema.org (`Physician`, `MedicalBusiness`, `MedicalClinic`, `Person`) et mapper :
  `name → nom`/`prenom`/`civilite` (découpage prudent), `medicalSpecialty`/`jobTitle → profession`,
  `address (PostalAddress) → adresse` + `arrondissement` (dérivé du code postal `75020` → `20e`),
  `geo (latitude/longitude) → latitude/longitude` (**+ `geocode_score` élevé** puisque source directe),
  l'URL canonique → `doctolib`, `sameAs`/`url` de cabinet → `site_web`, `knowsLanguage → langues`.
- **Repli DOM** (heuristique) uniquement pour les champs qu'aucun JSON-LD ne donne — sélecteurs ciblés,
  **tolérants à l'absence** (jamais d'exception si un nœud manque).
- **Patient-only** : ne mapper **que** des clés de la liste blanche S1. **Ne jamais** produire un champ
  pro (Doctolib n'en expose pas, et le lecteur S1 les rejetterait de toute façon — double barrière).
- `source_url` = l'URL de la page courante.

### Étapes
1. Créer `tools/doctolib-bookmarklet/extract.js` — **source lisible et commentée** (c'est elle qu'on
   maintiendra). Fonction `extractFromPage(document, location): Prefill`.
2. Encoder le résultat en **base64url** et construire `${APP_ORIGIN}/nouveau?prefill=<data>` ;
   `window.open(url, '_blank')`. `APP_ORIGIN` = l'URL Vercel de prod (constante en tête de fichier,
   **à renseigner**).
3. Champs manquants → simplement absents du payload (le formulaire les laisse vides). **Ne rien
   inventer** (doctrine T-005).

## T2 — Le bookmarklet + la notice

### Étapes
1. Produire le **bookmarklet** : la version minifiée `javascript:(()=>{…})()` de `extract.js`, dans
   `tools/doctolib-bookmarklet/bookmarklet.txt` (+ un petit script/commande de build documenté, ou une
   minification manuelle tracée — pas de chaîne de build lourde pour ~40 lignes).
2. **Notice d'installation** `tools/doctolib-bookmarklet/README.md` pour les ~10 membres :
   - glisser le lien dans la barre de favoris (30 s), captures/étapes ;
   - usage : ouvrir la fiche Doctolib du praticien → cliquer le favori → relire et enregistrer ;
   - rappel : **on relit toujours** (source déclarative, fiche « à vérifier »).
3. **Test CSP réel (Thibault)** : vérifier sur une vraie page Doctolib que le bookmarklet **s'exécute**.
   - S'il marche : consigner OK, S3 n'est pas déclenchée.
   - **S'il est bloqué par la CSP** de Doctolib : le **constater dans le bilan** → déclenche **S3**
     (mini-extension, même logique de lecture, non soumise à la CSP de la page). C'est le seul point qui
     décide de S3.

### Validation
- Auto : la logique de mapping de `extract.js` est **testable hors navigateur** (passer un
  `document`/HTML de fixture — un extrait de page Doctolib JSON-LD réel anonymisé). Si vitest en place,
  tester : JSON-LD complet → payload attendu ; JSON-LD partiel + repli DOM ; **aucun champ pro produit** ;
  page sans JSON-LD → payload minimal sans exception.
- Humain (réel) : Thibault teste sur ≥ 2 pages Doctolib (un praticien, un centre) → l'annuaire s'ouvre
  pré-rempli, données cohérentes, CSP OK/KO consigné.

### Si bloqué
JSON-LD absent ou structure inattendue sur les pages réelles → s'appuyer davantage sur le repli DOM
**pour les seuls champs sûrs**, et **documenter** ce que Doctolib expose réellement (ça oriente S3 et la
maintenance). Ne pas deviner un mapping incertain.
