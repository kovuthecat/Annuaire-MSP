## T1 — Lecteur de `prefill` (liste blanche + sécurité)

### Objectif
Transformer un `prefill` non fiable en un `FormState` sûr, sans jamais toucher à un champ pro.

### Décision clé
- **Liste blanche stricte** (cf. contrat de l'index) : seules les clés patient/identité/lieu sont lues ;
  toute autre clé (y compris un champ pro) est **ignorée silencieusement**.
- **Assainissement** : `String()` + `trim` + **borne de longueur** par champ ; rejet des valeurs non
  conformes (ex. `secteur_conv` hors `1|2|centre|non_conv` → ignoré). Jamais d'insertion HTML (les
  champs alimentent des `value` React contrôlées — pas de `dangerouslySetInnerHTML`).
- **Décodage tolérant** : base64url → JSON dans un `try/catch` ; payload illisible → on ouvre `/nouveau`
  **vide** (pas d'erreur bloquante) + un message discret « préremplissage illisible ».

### Étapes
1. Dans `formState.ts`, ajouter :
   - un type `Prefill` (les clés autorisées, toutes optionnelles) ;
   - `parsePrefill(raw: string): Prefill | null` — décode base64url, valide/liste-blanche/borne, renvoie
     `null` si irrécupérable ;
   - `formFromPrefill(p: Prefill): FormState` — part de `emptyForm()` et n'écrit **que** les champs
     présents et valides (patient/identité/lieu). `moyenDeContact` reste vide (les vraies coordonnées
     patient sont posées directement : `telSecretariat`, `emailRdv`, `doctolib`, `siteWeb`).
2. Prévoir le **transport de la provenance** jusqu'au payload de création : `source_url` (depuis le
   prefill), `source_type = 'doctolib'`, `statut = 'a_verifier'`. Comme `buildContactPayload` n'émet pas
   ces colonnes, ajouter un chemin dédié — au choix, le plus simple et le moins invasif :
   - étendre `ContactPayload` d'un supplément optionnel `{ source_url?, source_type?, statut? }` **posé
     uniquement en mode préremplissage**, ou
   - construire l'`insert` de provenance dans `EditionPage` par-dessus le payload standard.
   Le mode saisie manuelle **reste inchangé** (aucune provenance).
3. (Synergie P3, **optionnel et gracieux**) si le prefill porte `latitude`/`longitude`/`geocode_score`
   **et** que ces colonnes existent, les transporter aussi ; sinon les ignorer sans erreur.

## Bilan de session

### T1 — Lecteur de `prefill` (`src/features/edition/formState.ts`)

Ajouté après `formFromContact` (le fichier existant n'a pas été réécrit, seulement complété) :

- `export interface Prefill` — les 13 clés texte/enum de la liste blanche
  (`nom, prenom, civilite, profession, etablissement, adresse, arrondissement, doctolib, site_web,
  tel_secretariat, email_rdv, secteur_conv, langues`) + `source_url` + `latitude`/`longitude`/
  `geocode_score` (synergie P3), toutes optionnelles — reprend exactement le contrat figé dans
  `plans/P4/index.md`.
- `base64UrlDecode(raw)` — `-`/`_` → `+`/`/`, complément du padding, `atob`, puis `TextDecoder` UTF-8
  (nécessaire : les valeurs Doctolib contiennent des accents, un décodage `atob` brut les aurait
  corrompus). Toute exception (base64 invalide) remonte via `try/catch` de l'appelant.
- `parsePrefill(raw)` — décode + `JSON.parse` dans un seul `try/catch` → `null` si irrécupérable
  (base64 corrompu, JSON invalide, ou pas un objet). Sinon, ne lit **que** les clés de la liste
  blanche (toute autre clé, y compris un champ pro comme `portable`, est simplement absente du
  résultat — ignorée silencieusement, jamais d'erreur) ; chaque champ texte passe par `String()` +
  `trim()` + troncature à une borne dédiée (`PREFILL_MAX_LENGTHS`, 20 à 500 caractères selon le
  champ) ; `secteur_conv` est vérifié contre `1|2|centre|non_conv` (rejeté sinon) ;
  `latitude/longitude/geocode_score` acceptés seulement si `typeof === 'number' && Number.isFinite`.
- `formFromPrefill(p)` — part de `emptyForm()`, n'écrit que les champs présents. **Décision
  d'implémentation notable** (le contrat liste `nom`/`prenom`/`civilite` comme 3 clés séparées, mais
  `FormState` n'a qu'un seul champ `nom` — absence volontaire de `civilite`/`prenom` décidée en P1/S5
  T8) : `civilite` + `prenom` + `nom` sont composés dans le champ `nom` unique, dans cet ordre
  (« Dr Jean Dupont »), comme l'affichage Doctolib. `type` reste `'praticien'` (valeur par défaut
  d'`emptyForm()`, le contrat ne porte pas de clé `type`). `moyenDeContact` reste vide, comme demandé.
- **T1.2 (transport de la provenance)** : forme retenue = la **seconde option** proposée par le plan
  (« construire l'insert de provenance dans `EditionPage` par-dessus le payload standard »), **pas**
  d'extension de `ContactPayload`. Justification : `NewContactInput` (`src/data/directory.ts`) est
  déjà `Partial<ContactWritable>` et `ContactWritable` inclut `source_url`/`source_type`/`statut` —
  il suffit d'étaler `{ ...payload, source_url, source_type: 'doctolib', statut: 'a_verifier' }` au
  point d'appel de `createContact`, seulement quand un `prefill` valide est présent. `formState.ts`
  (donc le chemin de saisie manuelle) n'a **pas été touché** sur ce point : `ContactPayload` et
  `buildContactPayload` sont strictement inchangés, la provenance n'existe que côté `EditionPage`.
- **T1.3 (synergie P3, optionnel)** : **non câblé** intentionnellement. `Prefill` accepte bien
  `latitude`/`longitude`/`geocode_score` (contrat figé, prêt pour S2), mais rien ne les transporte
  vers l'insert aujourd'hui : ni `FormState` ni `ContactPayload` n'ont de colonnes dédiées au moment
  de la lecture initiale de `supabase/schema.sql` pour cette session. **Note de contexte** : en cours
  de session, une exécution concurrente d'un autre plan (P3) a introduit ces colonnes (`latitude`,
  `longitude`, `geocode_score`, `geocoded_at`) et un géocodage en tâche de fond après toute création
  avec adresse (indépendant de la provenance). Plutôt que de rebrancher T1.3 en pleine collision avec
  ce travail concurrent (hors périmètre de cette session, fichiers en cours d'écriture par l'autre
  agent), le choix a été de **laisser la dégradation gracieuse telle que prescrite par le plan** — le
  géocodage générique de P3 couvre de fait toute fiche créée avec une adresse, prefill ou non ; le
  gain d'un passage direct des coordonnées Doctolib (si présentes dans le JSON-LD) resterait une
  optimisation mineure (évite un appel géocodeur), à faire par une session ultérieure qui verra l'état
  stabilisé du schéma.

### T2 — Bandeau « pré-rempli depuis Doctolib » (`src/features/edition/EditionPage.tsx`)

- Import `useSearchParams` (react-router-dom v7) pour lire `?prefill=` — `useParams()` seul ne donne
  que les paramètres de route (`:id`), pas la query string.
- `rawPrefill` : lu **uniquement en création** (`mode === 'create'`) — en édition le paramètre est
  ignoré (une fiche existante n'est jamais réécrite par un prefill, décision explicite du plan).
- `prefill` (`useMemo` sur `rawPrefill`) : trois états distingués — `null` (pas de `prefill` dans
  l'URL), `undefined` (présent mais `parsePrefill` a renvoyé `null`, donc illisible),
  ou un `Prefill` assaini (potentiellement vide mais un objet, donc « valide »).
- État initial du formulaire (`useState` paresseux) : `formFromPrefill(prefill)` si `prefill` est un
  objet, sinon `emptyForm()` — donc `/nouveau` **sans** `prefill` prend exactement l'ancien chemin
  (`emptyForm()`), aucune régression sur le comportement existant.
- **Bandeau** (`prefillBannerStyle`, ton `colors.comment.info` — bleu sobre déjà utilisé pour le badge
  VAD, cohérent avec ARCHITECTURE §Ton visuel « pas de couleur notation agressive » donc pas la teinte
  ambre/alerte) : affiché si `prefill` est un objet (donc valide, même vide). Texte exact du plan :
  « Fiche pré-remplie depuis Doctolib — vérifiez chaque champ avant d'enregistrer. Source déclarative,
  non datée. »
- **Message discret** (`prefillUnreadableStyle`, `colors.text.muted`, pas de bordure ni de fond
  coloré) si `prefill === undefined` : « Préremplissage illisible — le formulaire s'est ouvert vide. »
  — jamais bloquant, le formulaire reste utilisable normalement (vide).
- **Soumission** (`handleSubmit`, mode création uniquement) : si `prefill` est un objet, l'insert
  envoyé à `createContact` est étendu par-dessus le `payload` standard avec
  `source_url: prefill.source_url ?? null`, `source_type: 'doctolib'`, `statut: 'a_verifier'` — sinon
  (saisie manuelle) le payload part strictement inchangé, sans aucune colonne de provenance.
- Encart de détection de doublon (S5) : **non touché**, reste branché sur `form.nom` comme avant —
  actif aussi bien pour une saisie manuelle que pour un nom prérempli.
