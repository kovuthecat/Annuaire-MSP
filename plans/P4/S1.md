## T1 — Lecteur de `prefill` (liste blanche + sécurité)

### Objectif
Transformer un `prefill` non fiable en un `FormState` sûr, sans jamais toucher à un champ pro.

### Décision clé
- **Liste blanche stricte** (cf. contrat de l'index) : seules les clés patient/identité/lieu sont lues ;
  toute autre clé (y compris un champ pro) est **ignorée silencieusement**.
- **Assainissement** : `String()` + `trim` + **borne de longueur** par champ ; rejet des valeurs non
  conformes (ex. `secteur_conv` hors `1|2|centre|non_conv` → ignoré). Jamais d'insertion HTML (les
  champs alimentent des `value` React contrôlées — pas de `dangerouslySetInnerHTML`).
- **Décodage tolérant** : base64url → JSON dans un `try/catch` ; payload illisible → on ouvre `/nouveau`
  **vide** (pas d'erreur bloquante) + un message discret « préremplissage illisible ».

### Étapes
1. Dans `formState.ts`, ajouter :
   - un type `Prefill` (les clés autorisées, toutes optionnelles) ;
   - `parsePrefill(raw: string): Prefill | null` — décode base64url, valide/liste-blanche/borne, renvoie
     `null` si irrécupérable ;
   - `formFromPrefill(p: Prefill): FormState` — part de `emptyForm()` et n'écrit **que** les champs
     présents et valides (patient/identité/lieu). `moyenDeContact` reste vide (les vraies coordonnées
     patient sont posées directement : `telSecretariat`, `emailRdv`, `doctolib`, `siteWeb`).
2. Prévoir le **transport de la provenance** jusqu'au payload de création : `source_url` (depuis le
   prefill), `source_type = 'doctolib'`, `statut = 'a_verifier'`. Comme `buildContactPayload` n'émet pas
   ces colonnes, ajouter un chemin dédié — au choix, le plus simple et le moins invasif :
   - étendre `ContactPayload` d'un supplément optionnel `{ source_url?, source_type?, statut? }` **posé
     uniquement en mode préremplissage**, ou
   - construire l'`insert` de provenance dans `EditionPage` par-dessus le payload standard.
   Le mode saisie manuelle **reste inchangé** (aucune provenance).
3. (Synergie P3, **optionnel et gracieux**) si le prefill porte `latitude`/`longitude`/`geocode_score`
   **et** que ces colonnes existent, les transporter aussi ; sinon les ignorer sans erreur.

## T2 — Bandeau « pré-rempli depuis Doctolib — à vérifier »

### Objectif
Que le membre **voie** que la fiche vient d'une source déclarative et doit être relue.

### Étapes
Dans `EditionPage.tsx` : si un `prefill` valide est présent (mode création), initialiser le formulaire
via `formFromPrefill` **au lieu de** `emptyForm`, et afficher un **bandeau** sobre en tête : « Fiche
pré-remplie depuis Doctolib — **vérifiez chaque champ** avant d'enregistrer. Source déclarative, non
datée. » (tokens existants, pas de rouge agressif — cf. ARCHITECTURE §Ton visuel). L'encart de
**détection de doublon** existant (S5) reste actif : un préremplissage peut viser un pro déjà en base.

### Validation
- Auto (gate) : `npm run build` + `npm run typecheck` → 0 erreur. Si vitest en place : tester
  `parsePrefill` (payload valide, champ pro ignoré, `secteur_conv` invalide rejeté, base64 corrompu →
  `null`, longueurs bornées).
- Humain (visuel) : checklist ci-dessous.

### Si bloqué
Écrire `source_url`/`statut` échoue à l'insert (`column …` / `check`) → le schéma n'est pas rejoué ou
une valeur viole une contrainte : **signaler**, ne pas retirer la provenance pour « faire passer ».
