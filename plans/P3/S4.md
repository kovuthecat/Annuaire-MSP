## T1 — Préparer le jeu d'arrêts (hors app)

### Objectif
Un fichier de données **petit et prêt à l'emploi**, borné géographiquement.

### Décision clé
- **Source** : open data Île-de-France Mobilités « arrêts » (data.iledefrance-mobilites.fr, licence
  ouverte). Télécharger **hors app** (aucune clé).
- **Bornage** : **Paris + communes limitrophes** (75 + première couronne immédiate), confirmé avec
  Thibault. Filtrer par emprise géographique (bbox) pour rester léger.
- **Élagage** : ne garder que `nom`, `lat`, `lng`, `modes` (métro/tram/bus/rer), `lignes` (indicatifs).
  **Regrouper** les arrêts homonymes proches (un même « Pelleport » a plusieurs quais) pour éviter le
  bruit.

### Étapes
1. Créer `supabase/import/transit_prep.py` (en-tête mode d'emploi) : lit le dump IDFM, filtre par bbox,
   élague, regroupe, écrit **`src/features/proximite/data/arrets.json`** (format compact).
2. **Maîtriser la taille** : viser un fichier raisonnable (idéalement < ~300–500 ko). Si trop gros,
   réduire l'emprise ou la précision des coordonnées (5 décimales suffisent). Documenter le poids final.

## T2 — Fonction « arrêts les plus proches »

### Étapes
Créer `src/features/proximite/transit.ts` (**pure, testable**) :
- `nearestStops(point: LatLng, opts?: { maxDistanceM?: number; limit?: number }): Stop[]` — parcourt
  `arrets.json`, calcule la distance (Haversine), trie, retient les plus proches dans un rayon
  (`maxDistanceM` défaut ~600 m, `limit` défaut ~5). Renvoie nom, modes, lignes, distance.
- **Chargement paresseux** du JSON (`import()` dynamique) pour ne pas alourdir le bundle initial :
  il n'est utile que sur la fiche.

## T3 — Bloc « Transports à proximité » sur la fiche

### Étapes
Dans `FichePage.tsx`, sous la carte (S3) ou le bloc coordonnées si S3 pas encore là : si la fiche a des
coordonnées, afficher les arrêts proches — ex. « **M** Pelleport · 200 m », « **Bus** 60, 64 · 150 m »,
« **T** T3b Porte de Bagnolet · 550 m ». Icônes/pastilles **sobres** (tokens). Aucun arrêt dans le rayon
→ ligne discrète « Aucun arrêt à moins de X m ». Pas de coordonnées → **ne pas afficher** le bloc.

### Validation
- Auto (gate) : `npm run build` + `npm run typecheck` → 0 erreur ; taille du bundle surveillée (le JSON
  doit être **code-splité**, pas dans le chunk initial). Si vitest en place : tester `nearestStops`
  (tri, rayon, limite) sur un mini-jeu.
- Humain (visuel) : checklist ci-dessous.

### Si bloqué
Jeu IDFM trop volumineux même borné → réduire l'emprise et **signaler** le compromis ; ne pas embarquer
un fichier de plusieurs Mo dans le bundle.
