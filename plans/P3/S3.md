## T1 — Dépendance + composant carte réutilisable

### Objectif
Un seul composant carte, utilisé par l'annuaire et la fiche.

### Décision clé
- **Deps runtime** : `leaflet` + `react-leaflet` (+ `@types/leaflet` en devDep). Importer le **CSS de
  Leaflet** (`leaflet/dist/leaflet.css`).
- **Piège connu des icônes** : le marqueur par défaut de Leaflet casse avec les bundlers (chemins
  d'images). **Utiliser un `divIcon` maison** (petit marqueur CSS/SVG) plutôt que l'icône PNG par
  défaut — évite le bug et reste dans le design system. Marqueurs distincts : **MSP** (repère fixe),
  **référence patient** (si active), **fiche(s)**.
- **Tuiles** : `TileLayer` OSM standard, avec l'**attribution obligatoire** « © OpenStreetMap
  contributors ». (Nuance RGPD assumée, cf. cadrage.)

### Étapes
1. `npm i leaflet react-leaflet` + `npm i -D @types/leaflet`. Vérifier que le **build ne gonfle pas
   déraisonnablement** ; si besoin, charger la carte en **import dynamique** (`React.lazy`) pour ne pas
   l'embarquer dans le bundle initial de l'annuaire.
2. Créer `src/components/Map/ProximityMap.tsx` :
   - props : `points: { id: string; coords: LatLng; label: string; kind: 'msp'|'reference'|'contact' }[]`,
     `activeId?`, `onSelect?(id)`, `fitToPoints?: boolean`.
   - rend une carte Leaflet, cadrée automatiquement sur l'ensemble des points (`fitBounds`), marqueurs
     par `kind`, popup au clic (label). Hauteur/rayon via tokens.
   - **Aucun point → ne pas afficher la carte** (ou message « position à préciser »).

## T2 — Carte partagée sur l'annuaire

### Objectif
Une carte unique montrant tous les résultats filtrés + la MSP + la référence active.

### Décision clé (perf)
**Une seule carte pour tout l'écran**, pas une mini-carte par ligne (cf. cadrage — 1000 cartes =
proscrit). La carte lit **les résultats déjà filtrés** (mêmes données que la liste).

### Étapes
1. Dans `AnnuairePage.tsx`, ajouter un **panneau carte repliable** (bouton « Afficher la carte ») qui
   passe à `ProximityMap` : les contacts filtrés **ayant des coordonnées**, + `MSP_COORDS`, + la
   référence active si ≠ MSP. Contacts sans coordonnées → **exclus de la carte** (comptés dans un petit
   « N sans position »).
2. **Lien carte ↔ liste** : `activeId`/`onSelect` — cliquer une épingle surligne/scrolle la ligne ;
   survoler/sélectionner une ligne peut centrer la carte (au minimum : épingle → ligne).
3. Replié par défaut sur mobile (la liste prime — cf. ARCHITECTURE §Contraintes UI).

### Lazy-load / bundle
Leaflet + react-leaflet sont chargés uniquement via `React.lazy` (`components/Map/index.ts`), donc
absents du bundle initial. Confirmé au build : chunk séparé `ProximityMap-*.js` (156 kB, gzip
~46 kB) + son CSS dédié, **non fusionné** dans le chunk principal. Le chunk principal
(`index-*.js`, ~539 kB, gzip ~157 kB) déclenche toujours l'avertissement Vite « chunk > 500 kB »,
mais ce poids est **pré-existant** (Supabase JS, react-router-dom, polices) et **indépendant** de
cette session — aucune contribution de Leaflet dedans (vérifié : le chunk carte est bien isolé).
Pas de `manualChunks` ajouté (hors périmètre de cette session, le lazy-loading suffisait à l'objectif
du plan).

### Écarts avec le plan
1. **`react-leaflet` pinné en 4.2.1, pas la dernière version (`^5`)** : `react-leaflet@5` exige
   React 19 (`peerDependencies react: ^19`), or le projet est en React 18.3.1 — `npm i` échouait en
   `ERESOLVE`. `react-leaflet@4.2.1` est la dernière version majeure compatible React 18 (peer
   `react: ^18.0.0`, `leaflet: ^1.9.0`) ; aucune fonctionnalité utilisée ici (`MapContainer`,
   `TileLayer`, `Marker`, `Popup`, `useMap`) n'est absente en v4. Pas d'invention : version choisie
   pour cause de compatibilité constatée, pas de contournement (`--force`/`--legacy-peer-deps`).
2. **Import nommé, pas `import L from 'leaflet'`** : `@types/leaflet` n'expose pas d'export par
   défaut (`export as namespace L` pour usage UMD global uniquement) ; utilisé
   `import { divIcon, latLngBounds } from 'leaflet'` à la place — fonctionnellement identique.
3. **Panneau carte annuaire replié par défaut sur desktop *et* mobile** (le plan ne l'exigeait
   explicitement que sur mobile) : simplification volontaire, pas de détection de viewport ajoutée —
   la liste prime dans les deux cas, cohérent avec l'esprit de la consigne.
4. **Lien carte → liste : épingle → ligne implémenté ; ligne → carte (survol/centrage) non
   implémenté** — le plan précise « au minimum : épingle → ligne », ce minimum est couvert ; le sens
   inverse est un ajout optionnel non fait ici (pas de blocage, juste un choix de périmètre).
5. **Distance sur la fiche : toujours par rapport à la MSP** (`MSP_COORDS`), pas à la référence
   active. Le plan offrait les deux options (« ou depuis la référence active ») ; j'ai retenu la
   formule littérale donnée en exemple pour rester cohérent avec les **deux points affichés sur la
   carte fiche** (fiche + MSP, sans 3ᵉ épingle référence) — afficher une distance « au patient » sans
   épingle patient sur cette carte aurait été incohérent visuellement.
6. Le compteur « N sans position » de l'annuaire compte les contacts **filtrés/triés** sans
   coordonnées (même liste que celle rendue), pas l'ensemble des contacts.
