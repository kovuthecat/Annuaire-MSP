## T2 — Géocodage BAN + backfill rejouable

### Objectif
Donner une position aux ~770 fiches qui ont une `adresse`, via la Base Adresse Nationale, avec un
**score de confiance** et une **trace**, sans jamais inventer.

### Décision clé
- **Source** : `https://api-adresse.data.gouv.fr/search/?q=<adresse>&limit=1` (sans clé). Passer
  l'`arrondissement`/le code postal en `postcode` quand il est connu pour **désambiguïser** (Paris est
  un nid d'homonymes d'odonymes). Utiliser le `score` renvoyé par la BAN tel quel.
- **Seuil** : `score ≥ 0,6` → on retient `latitude`/`longitude` ; **sinon on stocke quand même
  `geocode_score`** (utile à la relecture) mais **pas** de lat/lng → la fiche restera « position à
  préciser ». *Le seuil 0,6 est un point de départ : le rapport (T3) doit permettre de le réévaluer.*
- **Rejouable & idempotent** : ne re-géocoder que les fiches sans `geocoded_at` (ou dont l'adresse a
  changé). Throttling léger (la BAN est publique — rester poli, ~10-20 req/s max, backoff sur 429).
- **Défaut prudent** : en cas de doute (score limite, plusieurs candidats proches), **ne pas placer** —
  cohérent avec l'asymétrie du risque de T-005.

### Étapes
1. Créer `supabase/import/geocode.py` (en-tête « mode d'emploi » comme `cnam_join.py`) :
   - lit `annuaire_donnees.json`, itère sur les fiches ayant une `adresse` ;
   - construit la requête BAN (`q` = adresse ; `postcode` dérivé de l'arrondissement `75020` si
     dispo) ; retient le meilleur candidat + son `score` ;
   - écrit le résultat (`lat`, `lng`, `score`, `label` BAN retenu) **dans `_meta.geocode`** de chaque
     fiche (traçable, non importé en base — cohérent avec la façon dont `_meta` sert déjà de journal) ;
   - **rejouable** : saute les fiches déjà géocodées.
2. Produire le backfill applicable à la base : **`supabase/geo_backfill.sql`** — une série d'`update
   public.contacts set latitude=…, longitude=…, geocode_score=…, geocoded_at=now() where <clé>` **pour
   les seules fiches ≥ seuil**, plus les `update … set geocode_score=…, geocoded_at=now()` (sans
   lat/lng) pour les fiches < seuil.
   > **Clé de recollage à confirmer à l'exécution** : `annuaire_donnees.json` porte `idx`, mais les
   > lignes de `contacts` reçoivent un `uuid` au seed. **Lire `seed_annuaire.sql` pour savoir sur quoi
   > joindre** (probable : identité `nom`+`adresse`, ou une colonne de mapping). Si aucune clé stable
   > n'existe, l'alternative est de faire tourner le géocodage **après seed, directement sur la base**
   > (script avec la **service_role key** — jamais commitée — qui lit `id, adresse, arrondissement` et
   > `update` par `id`). **STOP et signaler le choix retenu** plutôt que deviner une clé fragile.

### Validation
- Auto : le SQL généré passe le parseur PostgreSQL ; le script est rejouable (2ᵉ run = 0 nouvelle
  requête). **Aucune** valeur lat/lng écrite pour un score < seuil.
- Humain : lire le rapport (T3), vérifier par sondage 5-10 fiches (label BAN cohérent avec l'adresse).

### Si bloqué
Clé de recollage introuvable → **ne pas inventer** ; proposer l'alternative « backfill post-seed sur la
base » et rendre la main. BAN indisponible / 429 persistant → throttle davantage, reprendre (le script
est rejouable), ne pas paralléliser à outrance.
